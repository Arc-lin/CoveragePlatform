## 1. 方案概述

本方案基于 **JaCoCo (Java Code Coverage)** 实现 Android 端代码覆盖率统计，支持自动收集、自动上传至覆盖率平台，并结合 Git Diff 进行增量覆盖率分析。

> ⚠️ **重要：创建 Build / 上传覆盖率报告时必须带上真实的 `commitHash`**
>
> 平台计算增量覆盖率的前提是：每份覆盖率报告都标注了它对应的**真实 git commit**。
> `commitHash` 不是一个可以随便填、填了就行的参数——它至少有三个地方依赖它：
>
> 1. **增量覆盖率计算**：平台用 `commitHash` 配合创建 Build / 上传报告时附带的 `gitDiff` 来确定
>    "这份报告对应哪次改动"，进而计算出改动行的覆盖情况。`commitHash` 不对应真实 commit，或者
>    每次都传同一个固定值/占位符，增量覆盖率就完全算不出来，平台只能退化成展示全量覆盖率。
> 2. **源码高亮展示**：`/source` 接口按 `commitHash` 去远程仓库（GitHub/GitLab/Gitee/Bitbucket）
>    拉取对应版本的源码叠加覆盖率标注，`commitHash` 不存在于远程仓库会导致拉取失败。
> 3. **历史趋势追踪**：覆盖率趋势图按 commit 时间线排列，固定/虚假的 `commitHash` 会让趋势失真。
>
> **接入时请务必：** 每次构建/上传都用 `git rev-parse HEAD`（或对应 CI 环境变量，如
> `${{ github.sha }}`）取真实 commit hash，不要写死成固定字符串。这样后台才能按 commit 维度
> 看清楚每次提交的覆盖率变化，而不是每次都是一份看不出关联关系的全量报告。

### 核心特性

| 特性 | 说明 |
|------|------|
| 🔄 自动收集 | App 进入后台 / 退出时自动 dump 覆盖率数据 |
| 📤 自动上传 | dump 后自动上传 `.ec` 文件至覆盖率平台 |
| 🔬 增量分析 | 配合 Git Diff，只统计变更代码的覆盖率 |
| 🛡 线程安全 | AtomicBoolean / AtomicLong + CAS 保证并发安全 |
| ⚡ 非阻塞 | IO 操作在 `Dispatchers.IO` 执行，避免主线程 ANR |
| 📦 批量上传 | 支持分批并发上传，每批最多 3 个文件 |

### 整体流程

```
编译时插桩(JaCoCo) → 运行时收集(探针) → 自动保存(.ec) → 自动上传(平台)
                                                              ↓
                                      CI 流程：XML报告 → Git Diff → 增量分析 → JSON报告
```

### 文件清单

| 文件 | 用途 |
|------|------|
| `CoverageCollector.kt` | 覆盖率数据收集器，管理生命周期回调与自动上传 |
| `CoverageUploader.kt` | 覆盖率数据上传器，基于 OkHttp 实现 |
| `incremental_coverage.py` | Python 增量覆盖率分析工具（CI 使用） |

## 2. 快速开始

### 2.0 在平台创建项目和 Build

**Step 1：创建项目**，拿到 `projectId`：

```bash
curl -X POST http://<平台地址>:3001/api/projects \
  -H "Content-Type: application/json" \
  -d '{
    "name": "你的项目名",
    "platform": "android",
    "repositoryUrl": "https://github.com/<owner>/<repo>.git"
  }'
```

私有仓库需要额外传 `accessToken`（GitHub PAT / GitLab PRIVATE-TOKEN / Gitee access_token /
Bitbucket App Password，按 `repositoryUrl` 域名对应平台生成）：

```bash
curl -X POST http://<平台地址>:3001/api/projects \
  -H "Content-Type: application/json" \
  -d '{
    "name": "你的项目名",
    "platform": "android",
    "repositoryUrl": "https://git.example.com/<owner>/<repo>.git",
    "accessToken": "<你的访问令牌>"
  }'
```

`accessToken` 是敏感凭据，平台只存不回显原文——查询时看到的是 `hasAccessToken: true/false`，
不是 token 原文；要更新就用 `PUT /api/projects/:id` 重新传一次。

**Step 2：创建 Build**，拿到 `buildId`（App 侧的 SDK 初始化只需要这一个 ID，不需要再传
`projectId`/`commitHash`/`branch`——这三个都在创建 Build 这一步定死了）：

```bash
curl -X POST http://<平台地址>:3001/api/builds \
  -F "binary=@app/build/intermediates/javac/devDebug/compileDevDebugJavaWithJavac/classfiles.zip" \
  -F "projectId=<上一步拿到的 PROJECT_ID>" \
  -F "commitHash=$(git rev-parse HEAD)" \
  -F "branch=$(git rev-parse --abbrev-ref HEAD)" \
  -F "gitDiff=$(git diff <base_commit>..HEAD --unified=0 -- app/src/main/java/)"
```

> `binary` 字段传的是编译后 `.class` 文件打包的 zip（JaCoCo 合并 `.ec` 时需要用它定位探针对应
> 的源码行），不是 APK。`gitDiff` 可选，不传就只有全量覆盖率、没有增量。

返回的 `data.id` 就是 `buildId`，记录下来配置到 App 里（见 2.3 节）。同一个 `buildId` 可以反复
接收多次上传，平台会自动合并、重新计算覆盖率（`rawUploadCount` 字段记录已上传次数）。

### 2.1 添加依赖

```gradle
dependencies {
    implementation 'com.squareup.okhttp3:okhttp:4.12.0'
    implementation 'org.jetbrains.kotlinx:kotlinx-coroutines-android:1.7.3'
}
```

### 2.2 开启覆盖率插桩

```gradle
android {
    buildTypes {
        debug {
            testCoverageEnabled true   // ⚠️ 必须开启
        }
    }
}

jacoco {
    toolVersion = "0.8.12"
}
```

### 2.3 Application 中初始化

```kotlin
class MyApplication : Application() {
    override fun onCreate() {
        super.onCreate()

        CoverageCollector.init(
            application = this,
            uploadConfig = CoverageUploader.UploadConfig(
                baseUrl = "https://coverage-platform.internal",
                buildId = "在平台创建 Build 后拿到的 buildId"
            )
        )
    }
}
```

以上三步完成即可实现**自动收集 + 自动上传**。App 进入后台时自动 dump 覆盖率数据并上传至平台。

## 3. Gradle 配置

### 3.1 根项目 build.gradle

```gradle
buildscript {
    dependencies {
        classpath 'org.jacoco:org.jacoco.core:0.8.12'
    }
}
```

### 3.2 Module build.gradle（完整示例）

```gradle
plugins {
    id 'com.android.application'
    id 'kotlin-android'
    id 'jacoco'
}

android {
    namespace 'com.example.app'

    buildTypes {
        debug {
            testCoverageEnabled true
        }
    }

    defaultConfig {
        // 从 CI 环境变量注入 Git 信息
        def gitCommitHash = System.getenv("GIT_COMMIT_HASH") ?: "unknown"
        def gitBranch = System.getenv("GIT_BRANCH") ?: "unknown"

        buildConfigField "String", "GIT_COMMIT_HASH", "\"$gitCommitHash\""
        buildConfigField "String", "GIT_BRANCH", "\"$gitBranch\""
    }
}

jacoco {
    toolVersion = "0.8.12"
}

dependencies {
    implementation 'com.squareup.okhttp3:okhttp:4.12.0'
    implementation 'org.jetbrains.kotlinx:kotlinx-coroutines-android:1.7.3'
}

// JaCoCo 单元测试报告 Task（用于 CI 增量分析）
task jacocoUnitTestReport(type: JacocoReport, dependsOn: ['testDebugUnitTest']) {
    reports {
        xml.required = true
        html.required = true
    }

    def fileFilter = [
        '**/R.class',
        '**/R$*.class',
        '**/BuildConfig.*',
        '**/Manifest*.*',
        '**/*Test*.*',
        'android/**/*.*'
    ]

    def debugTree = fileTree(
        dir: "${buildDir}/intermediates/javac/debug",
        excludes: fileFilter
    )
    def kotlinDebugTree = fileTree(
        dir: "${buildDir}/tmp/kotlin-classes/debug",
        excludes: fileFilter
    )

    classDirectories.setFrom(files([debugTree], [kotlinDebugTree]))
    sourceDirectories.setFrom(files(["src/main/java", "src/main/kotlin"]))
    executionData.setFrom(fileTree(dir: buildDir, includes: [
        // 单元测试产物
        'jacoco/testDebugUnitTest.exec',
        // 连接设备测试（无 productFlavor）
        // 'outputs/code_coverage/debugAndroidTest/connected/**/*.ec',
        // 连接设备测试（有 productFlavor，如 dev）— 按实际 flavor 名调整
        'outputs/code_coverage/devDebugAndroidTest/connected/**/*.ec'
    ]))
}
```

> **注意（有 productFlavor 时）**：若 build.gradle 定义了 `productFlavors`（如 `dev` / `prod`），`intermediates` 路径中会包含 flavor 名，完整路径示例：
> ```gradle
> dir: "${buildDir}/intermediates/javac/devDebug/compileDevDebugJavaWithJavac/classes"
> ```
> 若只有 `debug`/`release`，路径通常是 `intermediates/javac/debug/` 或 `intermediates/javac/debug/compileDebugJavaWithJavac/classes/`（AGP 版本不同有差异，建议实际编译后在 `build/intermediates/javac/` 下确认实际目录名）。

## 4. SDK 初始化

### 4.1 基本初始化

```kotlin
CoverageCollector.init(
    application = this,
    uploadConfig = CoverageUploader.UploadConfig(
        baseUrl = "https://coverage-platform.internal",
        buildId = "abc123-build-id"
    )
)
```

### 4.2 使用 InitConfig（结构化配置）

```kotlin
val config = CoverageCollector.InitConfig(
    uploadConfig = CoverageUploader.UploadConfig(
        baseUrl = "https://coverage-platform.internal",
        buildId = "abc123-build-id"
    ),
    autoUpload = true       // 默认 true，设为 false 则只 dump 不上传
)

CoverageCollector.init(application, config)
```

### 4.3 参数说明

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `application` | `Application` | ✅ | Application 实例 |
| `uploadConfig.baseUrl` | `String` | ✅ | 覆盖率平台地址。⚠️ Android 9+ 默认禁止明文 HTTP，生产环境请使用 `https://` |
| `uploadConfig.buildId` | `String` | ✅ | 在平台创建 Build 后拿到的 ID（见 2.0 节），`commitHash`/`branch`/`gitDiff` 都在创建 Build 时定，这里不需要再传 |
| `autoUpload` | `Boolean` | ❌ | 是否自动上传，默认 `true` |

### 4.4 Android 9+ HTTP 明文限制说明

Android 9（API 28）起默认禁止 HTTP 明文流量。如果你的平台服务部署在内网使用 HTTP，有两种解决方案：

**方案一（推荐）：使用 HTTPS**

```kotlin
baseUrl = "https://coverage-platform.internal"
```

**方案二：配置网络安全策略**

在 `res/xml/network_security_config.xml` 中：

```xml
<?xml version="1.0" encoding="utf-8"?>
<network-security-config>
    <domain-config cleartextTrafficPermitted="true">
        <domain includeSubdomains="true">coverage-platform.internal</domain>
    </domain-config>
</network-security-config>
```

并在 `AndroidManifest.xml` 中引用：

```xml
<application
    android:networkSecurityConfig="@xml/network_security_config"
    ...>
```

## 5. CI 自动创建 Build

新设计下 App 不再需要在编译期注入 git 信息（不用 `buildConfigField`/`BuildConfig` 那一套）——
`commitHash`/`branch`/`gitDiff` 都是 CI 在创建 Build 时直接传给平台的，App 只需要知道
`buildId`。典型 CI 流程：编译 → 创建 Build → 把 `buildId` 写进分发说明（或者打包进一个单独的
config 文件随测试包一起分发，不建议写死在源码里反复改）。

**Jenkins (Groovy Pipeline):**

```groovy
stage('Create Coverage Build') {
    steps {
        sh '''
            ./gradlew assembleDevDebug
            zip -r classfiles.zip app/build/intermediates/javac/devDebug/compileDevDebugJavaWithJavac/classes

            COMMIT_HASH=$(git rev-parse HEAD)
            BRANCH=$(git rev-parse --abbrev-ref HEAD)
            BASE_COMMIT=$(git merge-base origin/main HEAD)
            git diff $BASE_COMMIT HEAD --unified=0 -- app/src/main/java/ > /tmp/diff.txt

            curl -X POST ${COVERAGE_PLATFORM_URL}/api/builds \
              -F "binary=@classfiles.zip" \
              -F "projectId=${COVERAGE_PROJECT_ID}" \
              -F "commitHash=${COMMIT_HASH}" \
              -F "branch=${BRANCH}" \
              -F "gitDiff=</tmp/diff.txt" | tee build_response.json
        '''
    }
}
```

**GitHub Actions:**

```yaml
- name: Build and create coverage Build
  run: |
    ./gradlew assembleDevDebug
    zip -r classfiles.zip app/build/intermediates/javac/devDebug/compileDevDebugJavaWithJavac/classes

    BASE_COMMIT=$(git merge-base origin/main HEAD)
    git diff $BASE_COMMIT HEAD --unified=0 -- app/src/main/java/ > /tmp/diff.txt

    curl -X POST ${{ secrets.COVERAGE_PLATFORM_URL }}/api/builds \
      -F "binary=@classfiles.zip" \
      -F "projectId=${{ secrets.COVERAGE_PROJECT_ID }}" \
      -F "commitHash=${{ github.sha }}" \
      -F "branch=${{ github.ref_name }}" \
      -F "gitDiff=</tmp/diff.txt" | tee build_response.json
```

拿到 `build_response.json` 里的 `data.id`（`buildId`）后，配置进 App（见 4.1 节），再分发给
测试人员。

## 6. 覆盖率数据收集与上传

### 6.1 自动收集（推荐）

初始化后无需任何额外代码。SDK 通过 `ActivityLifecycleCallbacks` 自动监听：

- **App 进入后台**（所有 Activity paused）：自动 dump + 上传
- **App 退出**（Activity isFinishing）：补一次强制 dump + 上传

防重复机制：
- 最小 dump 间隔 **30 秒**
- 使用 CAS 原子操作防止并发 dump
- IO 操作在后台线程执行，不阻塞主线程

### 6.2 手动 dump

```kotlin
// 普通 dump（遵守 30 秒间隔限制）
val filePath = CoverageCollector.dumpCoverage(context)

// 强制 dump（忽略间隔限制）
val filePath = CoverageCollector.dumpCoverage(
    context = context,
    force = true,
    autoUpload = true      // 可选：覆盖初始化时的 autoUpload 设置
)
```

> **注意**：手动 `dumpCoverage()` 内部自动异步上传，无法直接获取上传结果。如需感知上传结果，请使用方式 6.3。

### 6.3 手动上传已有文件

```kotlin
lifecycleScope.launch {
    val latestFile = CoverageCollector.getLatestCoverageFile(context)
    if (latestFile != null) {
        val result = CoverageCollector.uploadCoverage(latestFile)
        if (result.success) {
            Log.d("Coverage", "上传成功: reportId=${result.reportId}")
        } else {
            Log.e("Coverage", "上传失败: ${result.message}")
        }
    }
}
```

### 6.4 批量上传

```kotlin
lifecycleScope.launch {
    val uploader = CoverageUploader.getInstance()
    val files = CoverageCollector.getCoverageFiles(context)

    val results = uploader.uploadMultiple(files)

    results.forEach { result ->
        Log.d("Upload", "success=${result.success}, message=${result.message}")
    }
}
```

> 批量上传每批最多 3 个并发，避免耗尽 OkHttp 连接池。

### 6.5 获取覆盖率文件

```kotlin
// 获取所有覆盖率文件（按时间倒序）
val files: List<File> = CoverageCollector.getCoverageFiles(context)

// 获取最新的覆盖率文件
val latest: File? = CoverageCollector.getLatestCoverageFile(context)

// 获取覆盖率文件目录路径
val dirPath: String = CoverageCollector.getCoverageDirPath(context)
// 返回: /data/data/<package_name>/files/coverage
```

### 6.6 通过 adb 提取覆盖率文件

```bash
# 获取包名下的覆盖率文件列表
adb shell run-as <package_name> ls files/coverage/

# 提取单个文件
adb shell run-as <package_name> cat files/coverage/coverage_20260615_132334_11364.ec > coverage.ec

# 批量提取
adb shell run-as <package_name> tar czf - files/coverage/ > coverage.tar.gz
```

### 6.7 清除覆盖率数据

```kotlin
CoverageCollector.clearCoverageData(context)
```

### 6.8 检查初始化状态

```kotlin
if (CoverageCollector.isInitialized()) {
    // SDK 已初始化
}

if (CoverageUploader.isInitialized()) {
    // 上传器已初始化
}
```

## 7. 增量覆盖率分析（CI 流程）

增量覆盖率分析在 CI 流水线中执行，只统计**本次变更代码**的覆盖率，而非全量。

### 7.1 生成 JaCoCo XML 报告

```bash
./gradlew jacocoUnitTestReport

# 报告输出位置（task 名为 jacocoUnitTestReport）
# app/build/reports/jacoco/jacocoUnitTestReport/jacocoUnitTestReport.xml
```

### 7.2 获取 Git Diff

```bash
# 获取两个 commit 之间的差异
git diff <old_commit> <new_commit> --unified=0 > diff.patch

# 或获取当前分支与主分支的差异
git diff origin/main...HEAD --unified=0 > diff.patch
```

### 7.3 运行增量分析

```bash
python3 incremental_coverage.py \
    --jacoco-report app/build/reports/jacoco/jacocoUnitTestReport/jacocoUnitTestReport.xml \
    --diff-file diff.patch \
    --output incremental-report.json \
    --old-commit abc123 \
    --new-commit def456 \
    --threshold 80.0
```

### 7.4 命令行参数

| 参数 | 简写 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `--jacoco-report` | `-j` | ✅ | — | JaCoCo XML 报告路径 |
| `--diff-file` | `-d` | ✅ | — | Git diff 文件路径 |
| `--output` | `-o` | ❌ | `incremental-coverage-report.json` | 输出报告路径 |
| `--old-commit` | | ❌ | — | 旧 commit hash |
| `--new-commit` | | ❌ | — | 新 commit hash |
| `--extensions` | `-e` | ❌ | `.java .kt` | 统计的文件扩展名 |
| `--threshold` | `-t` | ❌ | `80.0` | 覆盖率阈值（百分比） |

### 7.5 输出报告格式

```json
{
  "summary": {
    "old_commit": "abc123",
    "new_commit": "def456",
    "total_files": 5,
    "files_with_coverage": 4,
    "total_changed_lines": 120,
    "total_covered_lines": 96,
    "total_missed_lines": 24,
    "overall_coverage_percent": 80.0,
    "threshold": 80.0,
    "status": "PASS",
    "check_time": ""
  },
  "files": [
    {
      "file_path": "com/example/MainActivity.kt",
      "total_changed_lines": 30,
      "covered_lines": 28,
      "missed_lines": 2,
      "line_coverage_percent": 93.33,
      "changed_lines_detail": [
        {
          "line": 42,
          "covered": true,
          "instruction_covered": 3,
          "instruction_missed": 0
        },
        {
          "line": 56,
          "covered": false,
          "instruction_covered": 0,
          "instruction_missed": 5
        }
      ]
    }
  ],
  "generated_at": "/path/to/project"
}
```

### 7.6 CI 集成示例

**Jenkins Pipeline:**

```groovy
stage('Incremental Coverage') {
    steps {
        sh '''
            # 1. 生成 JaCoCo 报告
            ./gradlew jacocoUnitTestReport

            # 2. 获取 Git Diff
            git diff origin/main...HEAD --unified=0 > diff.patch

            # 3. 运行增量分析
            python3 incremental_coverage.py \
                --jacoco-report app/build/reports/jacoco/jacocoUnitTestReport/jacocoUnitTestReport.xml \
                --diff-file diff.patch \
                --output incremental-report.json \
                --threshold 80.0

            # 4. （可选）上传 JaCoCo XML 到覆盖率平台，用于在线查看覆盖率报告
            curl -X POST https://coverage-platform.internal/api/upload/coverage \
                -F "file=@app/build/reports/jacoco/jacocoUnitTestReport/jacocoUnitTestReport.xml" \
                -F "projectId=android-app" \
                -F "platform=android" \
                -F "commitHash=${GIT_COMMIT_HASH}" \
                -F "branch=${GIT_BRANCH}" \
                -F "gitDiff=<diff.patch"
        '''
    }
}
```

### jacocoUITestReport（UI/Espresso 测试，推荐）

Espresso 测试更贴近真实人工操作场景，覆盖率数据更有代表性：

```gradle
task jacocoUITestReport(type: JacocoReport) {
    group = "Reporting"
    description = "Generate JaCoCo coverage report from connected/instrumented tests"

    reports {
        xml.required = true
        html.required = true
    }

    def fileFilter = ['**/R.class', '**/R$*.class', '**/BuildConfig.*',
        '**/Manifest*.*', '**/*Test*.*', 'android/**/*.*', '**/coverage/**']

    classDirectories.setFrom(fileTree(
        dir: "${buildDir}/intermediates/javac/devDebug/compileDevDebugJavaWithJavac/classes",
        excludes: fileFilter
    ))
    sourceDirectories.setFrom(files(["src/main/java"]))
    executionData.setFrom(fileTree(dir: buildDir, includes: [
        'outputs/code_coverage/devDebugAndroidTest/connected/**/*.ec'
    ]))
}
```

运行：`./gradlew jacocoUITestReport`
输出：`app/build/reports/jacoco/jacocoUITestReport/jacocoUITestReport.xml`

## 9. 平台接口规范

### 9.1 上传覆盖率数据（CI/手动上传，需先生成 XML）

> ⚠️ **此接口只接受 JaCoCo XML 格式**，不支持直接上传 `.ec` 二进制文件。
> 请先运行 `./gradlew jacocoUnitTestReport` 或 `./gradlew jacocoUITestReport` 生成 XML，再上传。
> SDK 自动上传 `.ec` 文件走的是另一个接口：`POST /api/builds/:buildId/raw-coverage`（见 9.3）。

```
POST /api/upload/coverage
Content-Type: multipart/form-data
Header: X-API-Key: <api_key>  (可选)

表单字段：
  file         : File    JaCoCo XML 报告文件（.xml）
  projectId    : String  项目 ID
  platform     : String  固定 "android"
  commitHash   : String  Git commit hash
  branch       : String  Git 分支名
  gitDiff      : String  git diff 文本内容（可选，用于增量覆盖率分析）

成功响应 (200):
{
  "success": true,
  "message": "Upload successful",
  "reportId": "report_abc123"
}

失败响应:
{
  "success": false,
  "message": "错误描述"
}
```

### 9.2 创建 Build（SDK 自动上传的前提）

```
POST /api/builds
Content-Type: multipart/form-data

表单字段：
  binary       : File    classfiles.zip（编译后的 .class 文件打包，用于覆盖率计算）
  projectId    : String  项目 ID
  commitHash   : String  Git commit hash（⚠️ 必须是真实 commit，否则无法做增量覆盖率分析）
  branch       : String  Git 分支名
  gitDiff      : String  git diff 文本内容（可选，用于增量覆盖率分析）

成功响应 (200):
{
  "success": true,
  "data": {
    "buildId": "6a38f5a7b3ec204aa0cf9bbe"
  }
}
```

拿到的 `buildId` 就是 SDK 初始化时 `UploadConfig.buildId` 需要传入的值。

### 9.3 上传原始覆盖率数据（SDK 自动上传，直接传 `.ec`）

```
POST /api/builds/:buildId/raw-coverage
Content-Type: multipart/form-data

表单字段：
  file         : File    覆盖率原始文件（.ec，JaCoCo 二进制格式，SDK 自动生成）
  deviceInfo   : String  设备信息 JSON（可选，SDK 自动填写）
  testerName   : String  测试人员标识（可选）

成功响应 (200):
{
  "success": true,
  "message": "Raw coverage uploaded and merged successfully",
  "data": {
    "rawUploadId": "...",
    "buildId": "6a38f5a7b3ec204aa0cf9bbe",
    "reportId": "report_abc123",
    "rawUploadCount": 1,
    "coverage": { "...": "..." }
  }
}
```

服务端会按 `buildId` 自动合并同一个 Build 下的多次上传（多设备/多次测试），并基于创建 Build 时传入的 `commitHash`/`gitDiff` 计算增量覆盖率。`projectId`/`commitHash`/`branch` 不需要在这一步重复传入，因为已经在 9.2 创建 Build 时定死了。

## 10. 常见问题

### Q0: 覆盖率始终为 0%（All 0% coverage）

**原因**：使用了 AGP 8.x 的 `enableUnitTestCoverage = true`，这会启用 **offline instrumentation**，测试 JVM 加载的是未插桩的类，导致探针永远不触发。

**解决**：删除 `enableUnitTestCoverage true`，改用：

```gradle
android {
    buildTypes {
        debug {
            testCoverageEnabled true    // ✅ on-the-fly instrumentation，正确
            // enableUnitTestCoverage true  // ❌ 勿用，AGP 8.x 下导致 0%
        }
    }
}
```

`testCoverageEnabled = true` 在运行时注入 JaCoCo agent，`getExecutionData()` 才能拿到数据。

### Q1: `ClassNotFoundException: org.jacoco.agent.rt.RT`

**原因**：未开启 `testCoverageEnabled` 或 JaCoCo 版本不兼容。

**解决**：

```gradle
android {
    buildTypes {
        debug {
            testCoverageEnabled true    // 必须开启
        }
    }
}
jacoco {
    toolVersion = "0.8.12"
}
```

### Q2: dump 返回 null 或文件为空

**可能原因**：

1. App 未执行到任何被插桩的代码
2. JaCoCo agent 返回空数据
3. JaCoCo API 版本不匹配

**排查**：

```bash
adb logcat -s CoverageCollector:D
```

查看日志中的具体错误信息。

### Q3: 上传失败 `Network error`

**排查步骤**：

1. 检查 `baseUrl` 配置是否正确
2. 确认 App 有 `INTERNET` 权限
3. Android 9+ 检查是否使用 HTTPS 或配置了 `network_security_config`
4. 确认平台服务是否可访问
5. 查看详细日志：

```bash
adb logcat -s CoverageCollector:D CoverageUploader:D
```

### Q4: `CoverageUploader not initialized`

**原因**：调用上传方法前未初始化 `CoverageUploader`。

**解决**：确保 `CoverageCollector.init()` 中传入了 `uploadConfig`，或在调用上传前先调用 `CoverageUploader.initOnce()`。

### Q5: 增量覆盖率分析找不到文件

**原因**：Git diff 中的文件路径与 JaCoCo 报告中的路径不匹配。

**解决**：`incremental_coverage.py` 内置了智能路径匹配（文件名 + 包名匹配），大多数情况自动处理。如果仍然不匹配，检查：

1. 确保 JaCoCo 报告的生成配置正确（`sourceDirectories` 路径）
2. 确保 diff 使用 `--unified=0` 参数生成

### Q6: 如何只在 Debug 包中集成？

```gradle
dependencies {
    debugImplementation 'com.squareup.okhttp3:okhttp:4.12.0'
    debugImplementation 'org.jetbrains.kotlinx:kotlinx-coroutines-android:1.7.3'
}
```

在代码中使用反射或 `BuildConfig.DEBUG` 判断：

```kotlin
if (BuildConfig.DEBUG) {
    CoverageCollector.init(this, ...)
}
```

### Q7: 上传时如何添加自定义元数据？

```kotlin
lifecycleScope.launch {
    val uploader = CoverageUploader.getInstance()
    val result = uploader.uploadCoverage(
        coverageFile = file,
        commitHash = "abc123",
        branch = "main",
        metadata = mapOf(
            "tester" to "zhangsan",
            "testType" to "regression",
            "environment" to "staging"
        )
    )
}
```

### Q8: 如何查看覆盖率数据的保存位置？

```kotlin
val dir = CoverageCollector.getCoverageDirPath(context)
// → /data/data/<package_name>/files/coverage

// 文件命名格式: coverage_yyyyMMdd_HHmmss_<PID>.ec
// 示例: coverage_20260615_132334_11364.ec
```
