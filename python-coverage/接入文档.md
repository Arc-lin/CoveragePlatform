# Python 代码覆盖率接入文档

> 版本：v1.0

---

## 1. 方案概述

Python 项目使用 **`coverage.py`** 生成覆盖率报告，手动（或 CI 脚本）上传到平台。没有自动化采集方式
（移动端那种"App 退后台自动上传"对 Python 服务不适用），流程是：运行测试 → 生成报告文件 → 上传。

```
运行测试           生成报告              上传
(pytest/unittest) → (coverage xml/lcov/json) → (curl POST)
```

> ⚠️ **重要：上传覆盖率报告时必须带上真实的 `commitHash`**
>
> 平台计算增量覆盖率的前提是：每份覆盖率报告都标注了它对应的**真实 git commit**。
> `commitHash` 不是一个可以随便填、填了就行的参数——它至少有三个地方依赖它：
>
> 1. **增量覆盖率计算**：平台用 `commitHash` 配合上传时附带的 `gitDiff` 来确定"这份报告对应
>    哪次改动"，进而计算出改动行的覆盖情况。`commitHash` 不对应真实 commit，或者每次都传同一个
>    固定值/占位符，增量覆盖率就完全算不出来，平台只能退化成展示全量覆盖率。
> 2. **源码高亮展示**：`/source` 接口按 `commitHash` 去远程仓库（GitHub/GitLab/Gitee/Bitbucket）
>    拉取对应版本的源码叠加覆盖率标注，`commitHash` 不存在于远程仓库会导致拉取失败。
> 3. **历史趋势追踪**：覆盖率趋势图按 commit 时间线排列，固定/虚假的 `commitHash` 会让趋势失真。
>
> **接入时请务必：** 每次上传都用 `git rev-parse HEAD`（或对应 CI 环境变量，如
> `${{ github.sha }}`）取真实 commit hash，不要写死成固定字符串。这样后台才能按 commit 维度
> 看清楚每次提交的覆盖率变化，而不是每次都是一份看不出关联关系的全量报告。

平台支持 `coverage.py` 的三种输出格式：

| 格式 | 生成命令 | 文件扩展名 | 推荐度 |
|------|---------|-----------|--------|
| Cobertura XML | `coverage xml` | `.xml` | 推荐（信息最全） |
| LCOV | `coverage lcov` | `.info` | 可用 |
| JSON | `coverage json` | `.json` | 可用 |

> 不支持直接上传 `.coverage` 二进制文件，必须先转换为上述格式之一。

## 2. 前置准备

### 2.1 在平台注册项目

公开仓库：

```bash
curl -X POST http://<平台地址>:3001/api/projects \
  -H "Content-Type: application/json" \
  -d '{
    "name": "你的项目名",
    "platform": "python",
    "repositoryUrl": "https://github.com/<owner>/<repo>.git"
  }'
```

私有仓库需要额外传 `accessToken`（GitHub PAT / GitLab PRIVATE-TOKEN / Gitee access_token /
Bitbucket App Password，按 `repositoryUrl` 域名对应平台生成）：

```bash
curl -X POST http://<平台地址>:3001/api/projects \
  -H "Content-Type: application/json" \
  -d '{
    "name": "你的项目名",
    "platform": "python",
    "repositoryUrl": "https://git.example.com/<owner>/<repo>.git",
    "accessToken": "<你的访问令牌>"
  }'
```

返回示例：

```json
{
  "success": true,
  "data": {
    "id": "69a9aaef08b82d192c9907d3",
    "name": "MyService",
    "platform": "python",
    "repositoryUrl": "https://github.com/example/MyService.git",
    "hasAccessToken": false
  }
}
```

记录返回的 `id`（即 `PROJECT_ID`），后续上传需要。

> `repositoryUrl` 用于源码高亮展示，支持 GitHub / GitLab（含自建实例）/ Gitee / Bitbucket。
> `accessToken` 是敏感凭据，平台只存不回显原文——接口响应里看到的是 `hasAccessToken: true/false`，
> 不是 token 原文。后续如需修改，用 `PUT /api/projects/:id` 重新传 `accessToken` 即可覆盖。

### 2.2 安装依赖

```bash
pip install coverage pytest
```

> 如使用其他测试框架（unittest、nose2 等），`coverage run` 同样适用，不强制要求 pytest。

## 3. 接入步骤

### 3.1 运行测试并收集覆盖率

```bash
# 使用 pytest
coverage run -m pytest

# 或使用 unittest
coverage run -m unittest discover
```

如需指定源码目录（推荐），在 `.coveragerc` 中配置：

```ini
[run]
source = src/
branch = true
```

或在 `pyproject.toml`：

```toml
[tool.coverage.run]
source = ["src"]
branch = true
```

### 3.2 生成覆盖率报告

```bash
# 推荐：Cobertura XML
coverage xml -o coverage.xml

# 或 LCOV
coverage lcov -o coverage.info

# 或 JSON
coverage json -o coverage.json
```

### 3.3 上传到平台

```bash
curl -X POST http://<平台地址>:3001/api/upload/coverage \
  -F "file=@coverage.xml" \
  -F "projectId=<PROJECT_ID>" \
  -F "commitHash=$(git rev-parse HEAD)" \
  -F "branch=$(git rev-parse --abbrev-ref HEAD)"
```

### 3.4 带增量覆盖率上传

增量覆盖率衡量**本次变更的代码**被测试覆盖的程度，需要额外传 `gitDiff`：

```bash
# 生成 git diff（--unified=0 表示不带上下文行，平台解析增量覆盖率需要这个格式）
BASE_COMMIT=$(git merge-base origin/main HEAD)
git diff $BASE_COMMIT HEAD --unified=0 > /tmp/diff.txt

# 上传时附带 diff（注意 gitDiff=< 语法，< 前没有 @，是 curl 的文件内容引用）
curl -X POST http://<平台地址>:3001/api/upload/coverage \
  -F "file=@coverage.xml" \
  -F "projectId=<PROJECT_ID>" \
  -F "commitHash=$(git rev-parse HEAD)" \
  -F "branch=$(git rev-parse --abbrev-ref HEAD)" \
  -F "gitDiff=</tmp/diff.txt"
```

### 3.5 完整示例（Makefile）

```makefile
PLATFORM_URL = http://localhost:3001
PROJECT_ID = your_project_id

.PHONY: coverage upload

coverage:
	coverage run -m pytest
	coverage xml -o coverage.xml
	coverage report  # 终端显示摘要

upload: coverage
	@BASE=$$(git merge-base origin/main HEAD); \
	git diff $$BASE HEAD --unified=0 > /tmp/diff.txt; \
	curl -X POST $(PLATFORM_URL)/api/upload/coverage \
	  -F "file=@coverage.xml" \
	  -F "projectId=$(PROJECT_ID)" \
	  -F "commitHash=$$(git rev-parse HEAD)" \
	  -F "branch=$$(git rev-parse --abbrev-ref HEAD)" \
	  -F "gitDiff=</tmp/diff.txt"
```

### 3.6 CI/CD 集成（GitHub Actions）

```yaml
name: Python Coverage

on:
  push:
    branches: [ main, develop ]
  pull_request:

jobs:
  coverage:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 0

      - uses: actions/setup-python@v5
        with:
          python-version: '3.x'

      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install coverage pytest

      - name: Run tests with coverage
        run: |
          coverage run -m pytest
          coverage xml -o coverage.xml

      - name: Generate git diff
        run: |
          BASE_COMMIT=$(git merge-base origin/main HEAD)
          git diff $BASE_COMMIT HEAD --unified=0 > /tmp/diff.txt

      - name: Upload to Coverage Platform
        run: |
          curl -X POST ${{ secrets.COVERAGE_PLATFORM_URL }}/api/upload/coverage \
            -F "projectId=${{ secrets.PROJECT_ID }}" \
            -F "commitHash=${{ github.sha }}" \
            -F "branch=${{ github.ref_name }}" \
            -F "file=@coverage.xml" \
            -F "gitDiff=</tmp/diff.txt"
```

## 4. API 参考

### 项目管理

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/projects` | 创建项目 |
| `GET` | `/api/projects` | 获取所有项目 |
| `GET` | `/api/projects/:id` | 获取单个项目 |
| `PUT` | `/api/projects/:id` | 更新项目（包括 `accessToken`） |
| `DELETE` | `/api/projects/:id` | 删除项目 |

### 覆盖率上传与查询

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/upload/coverage` | 上传覆盖率报告文件 |
| `GET` | `/api/coverage/project/:projectId` | 获取项目的所有报告 |
| `GET` | `/api/coverage/project/:projectId/latest` | 获取最新报告 |
| `GET` | `/api/coverage/:id` | 获取报告详情 |
| `GET` | `/api/coverage/:id/files` | 获取文件列表 |
| `GET` | `/api/coverage/:id/incremental` | 获取增量覆盖率详情 |
| `GET` | `/api/coverage/:id/source?path=<filePath>` | 获取源码（支持 GitHub/GitLab/Gitee/Bitbucket） |

## 5. 常见问题

### Q: 同一份覆盖率能多次上传吗？

可以，每次上传都会生成一条新的 CoverageReport 记录（不会像移动端 Build 那样自动合并多次原始数据），
适合按 commit/CI 流水线粒度独立追踪覆盖率历史趋势。

### Q: 增量覆盖率显示为 "-"？

1. 上传时没有传 `gitDiff` 参数
2. `gitDiff` 内容为空（没有代码变更）
3. 变更的文件不在覆盖率报告中（如只改了配置文件）

### Q: 源码查看页面显示 "Source file not found"？

1. 确认项目的 `repositoryUrl` 配置正确（完整 URL，含 host/owner/repo）
2. 确认代码已 push 到远程仓库（GitHub/GitLab/Gitee/Bitbucket 均可）
3. 确认 `commitHash` 对应的 commit 存在于远程仓库
4. 如果是私有仓库，确认项目已配置 `accessToken`

### Q: Python 覆盖率只统计了部分文件？

在 `.coveragerc` 或 `pyproject.toml` 中配置 `source` 参数，确保覆盖率收集范围正确（见 2.2 节）。

### Q: Python 项目的增量覆盖率不准确？

确保 Cobertura XML 中的文件路径与 git diff 中的路径能正确匹配。如果 `.coveragerc` 中配置了
`source = src/`，则 XML 中文件名可能只是 `calculator.py`（不含 `src/` 前缀），而 diff 中是
`src/calculator.py`。平台会自动做模糊匹配，但项目结构复杂时建议保持一致的路径风格。

### Q: 方法覆盖率（functionCoverage）一直是 0？

`coverage.py` 标准的 Cobertura XML 输出**不会**填充 `<method line-rate="...">` 这个属性（`<methods>`
节点永远是空的），所以平台解析出来的 functionCoverage 对 Python 项目来说基本都是 0，这是
`coverage.py` 工具本身的限制，不是平台 bug。如果需要方法级覆盖率，需要用其它能生成这部分数据的工具
（如某些 Java/Swift 生态的工具），或者直接看 `coverage report -m` 的终端输出。

### Q: 服务端报错 "Failed to parse coverage file"？

确认上传的文件格式和 `platform=python` 匹配（Cobertura XML / LCOV `.info` / JSON 之一），不要
直接上传 `coverage run` 生成的 `.coverage` 二进制文件。
