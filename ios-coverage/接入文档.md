## 1. 方案概述

iOS 端采用 **LLVM Profile Instrumentation**（`-fprofile-instr-generate -fcoverage-mapping`，和
Xcode "Gather coverage for" 用的是同一套机制）进行代码覆盖率统计，配合 Git Diff 实现增量覆盖率分析。

> ⚠️ **重要：上传覆盖率报告时必须带上真实的 `commitHash`**
>
> 平台计算增量覆盖率的前提是：每份覆盖率报告都标注了它对应的**真实 git commit**。
> `commitHash` 不是一个可以随便填、填了就行的参数——它至少有三个地方依赖它：
>
> 1. **增量覆盖率计算**：平台用 `commitHash` 配合上传时附带的 `gitDiff` 来确定"这份报告对应
>    哪次改动"，进而计算出改动行的覆盖情况。`commitHash` 不对应真实 commit，或者每次都传同一个
>    固定值/占位符，增量覆盖率就完全算不出来，平台只能退化成展示全量覆盖率。
> 2. **源码高亮展示**：`/source` 接口按 `commitHash` 去远程仓库（GitHub/GitLab/Gitee/Bitbucket）
>    拉取对应版本的源码叠加覆盖率标注，`commitHash` 不存在于远程仓库会导致拉取失败。
> 3. **历史趋势追踪**：覆盖率趋势图按 commit 时间线排列，固定/虚假的 `commitHash` 会让趋势失真。
>
> **接入时请务必：** 每次构建/上传都用 `git rev-parse HEAD`（或对应 CI 环境变量，如
> `${{ github.sha }}`）取真实 commit hash，不要写死成固定字符串，也不要用 `git describe`、
> 版本号等代替。这样后台才能按 commit 维度看清楚每次提交的覆盖率变化，而不是每次都是一份
> 看不出关联关系的全量报告。

### 核心原理

```
编译时插桩                运行时导出                  数据合并            报告生成
(Clang/Swift flags)  →  (__llvm_profile_write_file)  →  (.profdata)  →  (lcov/HTML)
                                                                            ↓
                                              CI 流程：Git Diff → diffParser.rb → 增量报告
```

| 阶段 | 技术 | 说明 |
|------|------|------|
| 编译时插桩 | `-fprofile-instr-generate -fcoverage-mapping`（C/OC）/ `-profile-generate -profile-coverage-mapping`（Swift） | 编译器在代码中插入覆盖率计数器 |
| 数据导出 | `__llvm_profile_write_file()` | App 运行时把内存里的计数器导出为 `.profraw` 原始数据 |
| 数据合并 | `llvm-profdata merge` | 将多个 `.profraw` 合并为 `.profdata` |
| 报告生成 | `llvm-cov export` + `genhtml` | 导出 lcov 格式并生成 HTML 报告 |

### 与 Android 方案对比

| 特性 | iOS | Android |
|------|-----|---------|
| 插桩工具 | LLVM Profile Instrumentation / Clang | JaCoCo |
| 数据格式 | `.profraw` → `.profdata` | `.ec` |
| 报告工具 | `llvm-cov` + `genhtml` | JaCoCo XML |
| 自动上传 | `CoverageUploader`（`NSURLSession`） | `CoverageUploader`（OkHttp） |
| 手动上传 | Shell 脚本 / cURL | cURL |
| 增量分析 | 平台按 Build 的 `gitDiff` 自动计算（也可用 Ruby `diffParser.rb` 本地分析） | 平台按 Build 的 `gitDiff` 自动计算（也可用 Python `incremental_coverage.py` 本地分析） |

### 文件清单

| 文件 | 用途 |
|------|------|
| `CoverageCollector.h` | Objective-C 头文件（覆盖率落盘，对外 API） |
| `CoverageCollector.m` | Objective-C 实现（LLVM Profile 调用） |
| `CoverageUploader.h` | Objective-C 头文件（覆盖率上传，对外 API） |
| `CoverageUploader.m` | Objective-C 实现（`POST /api/builds/:buildId/raw-coverage`） |
| `compute_build_key.sh` | Shell 脚本：计算"构建身份"（单仓库=commit hash，组件化=复合指纹），Xcode Build Phase 和 CI 都要调用同一份（见第 6.0 节） |
| `extract_component_repos.sh` | Shell 脚本：提取 git 源组件清单（仓库地址 + 锁定 commit），CI 建 Build 时传给平台，用于壳工程仓库拉不到文件时的备查（见第 6.0/10.1 节） |
| `coverage_report.sh` | Shell 脚本：一键生成全量 / 增量覆盖率报告（手动方式用） |
| `gitdiff/utils/diffParser.rb` | Ruby 脚本：增量覆盖率解析（Git Diff → lcov gather，手动方式用） |

> 只提供 Objective-C 实现，没有单独的 Swift 版本。Swift 项目通过 Bridging Header 混编直接调用
> `CoverageCollector`，见 [5. Swift 项目接入](#5-swift-项目接入)。

## 2. 快速开始

### 2.0 在平台注册项目

接入前先在平台创建项目，拿到 `projectId`（下文构建产物上传、覆盖率上传都需要）：

```bash
curl -X POST http://<平台地址>:3001/api/projects \
  -H "Content-Type: application/json" \
  -d '{
    "name": "你的项目名",
    "platform": "ios",
    "repositoryUrl": "https://github.com/<owner>/<repo>.git"
  }'
```

私有仓库需要额外传 `accessToken`（GitHub PAT / GitLab PRIVATE-TOKEN / Gitee access_token /
Bitbucket App Password，按 `repositoryUrl` 域名对应平台生成）：

```bash
curl -X POST http://<平台地址>:3001/api/projects \
  -H "Content-Type: application/json" \
  -d '{
    "name": "你的项目名",
    "platform": "ios",
    "repositoryUrl": "https://git.example.com/<owner>/<repo>.git",
    "accessToken": "<你的访问令牌>"
  }'
```

返回的 `data.id` 就是后续要用的 `projectId`。`accessToken` 是敏感凭据，平台只存不回显原文——
查询时看到的是 `hasAccessToken: true/false`，不是 token 原文；要更新就用 `PUT /api/projects/:id`
重新传一次。

### 三步集成

**Step 1：添加编译参数**

在 Xcode Build Settings 中添加：

```
Other C Flags:      -fprofile-instr-generate -fcoverage-mapping
Other Swift Flags:  -profile-generate -profile-coverage-mapping   （纯 Swift 工程才需要，见第 5 节）
Other Linker Flags: -fprofile-instr-generate
```

**Step 2：添加源文件**

将 `CoverageCollector.h` 和 `CoverageCollector.m` 拖入 Xcode 项目。

**Step 3：AppDelegate 中初始化**

```objc
// AppDelegate.m
#import "CoverageCollector.h"

- (BOOL)application:(UIApplication *)application 
    didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {
    // 初始化覆盖率收集器
    [CoverageCollector initializeCollector];
    return YES;
}

- (void)applicationDidEnterBackground:(UIApplication *)application {
    // App 进入后台时保存覆盖率数据
    [CoverageCollector dumpCoverageData];
}

- (void)applicationWillTerminate:(UIApplication *)application {
    // App 即将终止时保存覆盖率数据
    [CoverageCollector dumpCoverageData];
}
```

完成以上三步后，运行 App 即可自动收集覆盖率数据（落盘到 `Documents/Coverage/`）。

> 上面三步只做到"本地落盘"。如果要做到"测试人员退后台自动上传到平台"（对标 Android 的能力），
> 还需要接入 `CoverageUploader`，见 [6. App 生命周期集成](#6-app-生命周期集成)。

## 3. Xcode 编译配置

### 3.1 Objective-C / C / C++ 文件

在 **Build Settings** 中搜索并配置以下项：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| **Other C Flags** | `-fprofile-instr-generate -fcoverage-mapping` | 启用 Clang 代码覆盖率插桩 |
| **Other Linker Flags** | `-fprofile-instr-generate` | 链接覆盖率运行时库 |

```
-fprofile-instr-generate -fcoverage-mapping
```

> 这是 Apple 推荐的覆盖率方案（和 Xcode "Gather coverage for" 用的是同一套机制），配合
> `llvm-cov` 工具链使用，兼容性好。`CoverageCollector` 也是基于这套机制实现的，不要和
> [LLVM Sanitizer Coverage](https://clang.llvm.org/docs/SanitizerCoverage.html)（`-sanitize-coverage=...`，
> 一套完全不同的、基于 `__sanitizer_cov_trace_pc_guard` 回调的插桩机制）搞混——两者互不兼容，
> `CoverageCollector` 没有用到后者。

### 3.2 Swift 文件（混编工程才需要）

纯 OC 工程跳过这一节。如果工程里有 Swift 代码（混编），在 **Build Settings** 中额外配置：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| **Other Swift Flags** | `-profile-generate -profile-coverage-mapping` | Swift 文件的覆盖率插桩，和 OC 的 `-fprofile-instr-generate` 是同一套机制对应的 Swift 编译器参数 |

### 3.3 仅 Debug 配置启用（推荐）

为避免 Release 包体积增大和性能影响，**且不要直接改全局 Debug 配置**——插桩会拖慢日常开发的每次
增量编译。推荐单独建一个 Build Configuration（例如复制一份 Debug，命名为 `Debug-Coverage`），只在
这个专用 Configuration 上填写编译参数：

1. 在 Xcode 项目设置里复制 Debug Configuration，命名为 `Debug-Coverage`
2. 在 **Build Settings** 里只在 `Debug-Coverage` 这一行填写上述编译参数，原来的 `Debug`/`Release` 都留空
3. 跑覆盖率测试时切换 Scheme 的 Build Configuration 到 `Debug-Coverage`

### 3.4 配置截图参考

```
Build Settings:
├── Other C Flags
│   ├── Debug:          (空)
│   ├── Debug-Coverage: -fprofile-instr-generate -fcoverage-mapping
│   └── Release:        (空)
├── Other Swift Flags（混编工程）
│   ├── Debug:          (空)
│   ├── Debug-Coverage: -profile-generate -profile-coverage-mapping
│   └── Release:        (空)
└── Other Linker Flags
    ├── Debug:          (空)
    ├── Debug-Coverage: -fprofile-instr-generate
    └── Release:        (空)
```

### 3.5 验证编译参数是否生效

编译后在 Xcode Build Log 中搜索 `-fprofile-instr-generate`，确认编译命令中包含该参数。

## 4. SDK 集成（Objective-C）

### 4.1 文件引入

将以下文件添加到 Xcode 项目中：

- `CoverageCollector.h` / `.m`（覆盖率落盘）
- `CoverageUploader.h` / `.m`（覆盖率上传，见 [6. App 生命周期集成](#6-app-生命周期集成)）

### 4.2 头文件 API

```objc
// CoverageCollector.h
@interface CoverageCollector : NSObject

/// 初始化覆盖率收集器，设置输出文件路径
/// 建议在 Application 启动时调用
+ (void)initializeCollector;

/// 手动触发覆盖率数据保存
/// @return 是否保存成功
+ (BOOL)dumpCoverageData;

/// 获取覆盖率数据文件路径
+ (nullable NSString *)coverageFilePath;

/// 获取所有历史覆盖率文件
+ (NSArray<NSString *> *)allCoverageFiles;

/// 清除所有覆盖率数据
+ (void)clearCoverageData;

/// 设置覆盖率数据输出目录（可选，默认为 Document/Coverage 目录）
+ (void)setCoverageDirectory:(NSString *)directory;

@end
```

### 4.3 实现原理

`CoverageCollector.m` 核心实现是对 LLVM Profile 运行时函数的硬 `extern` 声明 + 直接调用：

```objc
// 设置输出文件路径
__llvm_profile_set_filename([coveragePath UTF8String]);

// 写入覆盖率数据到文件
int result = __llvm_profile_write_file();
// result == 0 表示成功
```

### 4.4 自定义输出目录

```objc
// 设置自定义输出目录
NSString *customDir = [NSTemporaryDirectory() stringByAppendingPathComponent:@"MyCoverage"];
[CoverageCollector setCoverageDirectory:customDir];
```

默认输出目录为 `Documents/Coverage/`。

## 5. Swift 项目接入

`CoverageCollector` 没有独立的 Swift 实现，Swift 项目通过 **Bridging Header 混编**直接调用
Objective-C 版本。

### 5.1 接入步骤

1. 把 `CoverageCollector.h`、`CoverageCollector.m` 拖进 Xcode 工程，加入 App Target
2. 配置 Bridging Header（Build Settings 里的 "Objective-C Bridging Header"，没有的话新建一个
   `.h` 文件指过去），在里面 `#import "CoverageCollector.h"`：

```objc
// YourApp-Bridging-Header.h
#import "CoverageCollector.h"
```

3. Swift 代码直接调用，跟调用任何其它 Objective-C 类一样：

```swift
// AppDelegate.swift
import UIKit

@main
class AppDelegate: UIResponder, UIApplicationDelegate {

    func application(_ application: UIApplication,
                     didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
        CoverageCollector.initializeCollector()
        return true
    }

    func applicationDidEnterBackground(_ application: UIApplication) {
        CoverageCollector.dumpCoverageData()
    }

    func applicationWillTerminate(_ application: UIApplication) {
        CoverageCollector.dumpCoverageData()
    }
}
```

不需要任何额外胶水代码——Objective-C 的 `+ (void)initializeCollector` 在 Swift 里就是
`CoverageCollector.initializeCollector()`，命名按 Swift 的 Objective-C 桥接规则自动转换。

## 6. App 生命周期集成

### 6.0 前置步骤：让构建身份自动跟着包走

`CoverageUploader` 不需要手动维护 `buildId`——只需要传 `baseURL` + `projectId`，"构建身份"
（`buildKey`）会自动从编译时生成的 `CoverageInfo.plist` 读取，首次上传时用
`(projectId, buildKey)` 换成 `buildId`。这样改一次代码、重新打个包，不需要再去改源码里写死
的 ID。

> **单仓库项目**：`buildKey` 就是壳工程的完整 commit hash，跟字面意思一样。
>
> **CocoaPods + git 源组件化项目**：如果某个组件升级了版本但壳工程自己的 commit 没变，单纯用
> commit hash 当身份会把两次完全不同的构建误判成同一个——`buildKey` 改成壳工程 commit 和所有
> git 源组件锁定的 commit（从 `Podfile.lock` 的 `CHECKOUT OPTIONS` 段解析）拼起来算的复合
> 指纹，组件升级了指纹就会变。`compute_build_key.sh` 这个脚本会自动判断：检测不到 git 源组件就
> 直接退化成普通 commit hash，单仓库项目不需要任何额外配置。

**Step 1：把 `compute_build_key.sh` 拖进工程**（跟 `CoverageCollector.h/.m` 放一起），然后在
Xcode 里给 App Target 新增一个 Run Script Build Phase（放在所有 Build Phase 最后，确保在
CodeSign 之前执行）：

```bash
BUILD_KEY=$("${SRCROOT}/Coverage/compute_build_key.sh" "${SRCROOT}")
PLIST_PATH="${BUILT_PRODUCTS_DIR}/${UNLOCALIZED_RESOURCES_FOLDER_PATH}/CoverageInfo.plist"
cat > "${PLIST_PATH}" <<PLIST
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
	<key>GitCommitHash</key>
	<string>${BUILD_KEY}</string>
</dict>
</plist>
PLIST
echo "[Coverage] Wrote GitCommitHash(buildKey)=${BUILD_KEY} to ${PLIST_PATH}"
```

> plist 里的 key 还是叫 `GitCommitHash`（保持跟 `CoverageUploader` 读取的字段名一致，SDK 代码
> 不需要因为组件化而改），单仓库项目里它的值就是真实 commit；组件化项目里它的值是复合指纹——
> `CoverageUploader` 不关心这个区别，原样读出来传给平台就行。

每次编译都会重新生成这个 `CoverageInfo.plist`，本地编译和 CI 编译走的是**同一份**
`compute_build_key.sh` 脚本，不需要额外区分环境——这一点很重要，下一步 CI 创建 Build 时也必须
调用同一份脚本算出同一个值，两边任何一边单独改算法都会导致对不上。

**Step 2：CI（或本机）用 `compute_build_key.sh` 算出的同一个值创建 Build**，上传编译产物：

```bash
xcodebuild build -workspace YourApp.xcworkspace -scheme YourApp -configuration Debug \
  -destination 'generic/platform=iOS' -derivedDataPath ./build

BINARY=./build/Build/Products/Debug-iphoneos/YourApp.app/YourApp
BUILD_KEY=$(./compute_build_key.sh .)
COMPONENT_REPOS=$(./extract_component_repos.sh .)

curl -X POST https://coverage-platform.internal/api/builds \
  -F "binary=@${BINARY}" \
  -F "projectId=<PROJECT_ID>" \
  -F "commitHash=$(git rev-parse HEAD)" \
  -F "buildKey=${BUILD_KEY}" \
  -F "branch=$(git rev-parse --abbrev-ref HEAD)" \
  -F "gitDiff=$(git diff <base_commit>..HEAD --unified=0 -- YourApp/)" \
  -F "componentRepos=${COMPONENT_REPOS}"
```

> `commitHash` 永远传壳工程真实的 commit（平台拉源码、展示用，语义不变）；`buildKey` 不传的话
> 平台会自动用 `commitHash` 当默认值——单仓库项目可以直接不传这两个参数，跟以前的接入方式完全
> 一样。`componentRepos` 是平台拉源码时的"备查清单"：某个文件壳工程仓库里拉不到，会依次去这份
> 清单里列的组件仓库、按各自记录的 commit 再试一次——前端代码面板上会显示"来源：组件名"标签。

> ⚠️ 同一个 commit 被重复构建多次（比如换签名重新打包）没关系——平台按 `(projectId, commitHash)`
> 复用同一个 Build，多个 IPA 最终会对应同一个 buildId，覆盖率数据照常合并，不会产生重复记录。
>
> 这一步是硬性前提：App 只知道 commitHash，换不出 buildId 就说明这个 commit 还没有匹配的编译
> 产物可以解析覆盖率数据——这种情况下 `CoverageUploader` 会跳过上传并打日志，不会崩溃或重试风暴。

### 6.1 推荐触发时机

| 触发时机 | 方法 | 说明 |
|----------|------|------|
| App 启动 | `initializeCollector` + `initWithBaseURL:projectId:` | 设置 `.profraw` 输出路径，初始化上传器 |
| 进入后台 | `dumpCoverageData` → `uploadLatestCoverageFileWithCompletion:` | 落盘后立即上传到平台 |
| 即将终止 | `dumpCoverageData` → `uploadLatestCoverageFileWithCompletion:` | 确保退出前数据不丢失且已上传 |

> `CoverageUploader` 走 `POST /api/builds/:buildId/raw-coverage`，但不需要 App 直接知道
> `buildId`——首次上传时会自动用 `commitHash` 换出来并缓存。一个 `buildId` 可以反复接收多次
> 上传，平台会自动合并、重新计算覆盖率——这正是"测试人员正常用 App，退后台自动上传"这个场景
> 需要的能力。

### 6.2 完整 AppDelegate 示例

**Objective-C:**

```objc
#import "CoverageCollector.h"
#import "CoverageUploader.h"

@implementation AppDelegate

- (BOOL)application:(UIApplication *)application 
    didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {
    // 可选：设置自定义输出目录
    // [CoverageCollector setCoverageDirectory:@"/path/to/coverage"];

    [CoverageCollector initializeCollector];
    NSLog(@"[Coverage] 初始化完成，输出路径: %@", [CoverageCollector coverageFilePath]);

    [CoverageUploader initWithBaseURL:@"https://coverage-platform.internal"
                              projectId:@"<在平台创建项目后拿到的 projectId>"];
    return YES;
}

- (void)applicationDidEnterBackground:(UIApplication *)application {
    BOOL success = [CoverageCollector dumpCoverageData];
    NSLog(@"[Coverage] 后台保存%@", success ? @"成功" : @"失败");
    if (success) {
        [CoverageUploader uploadLatestCoverageFileWithCompletion:^(BOOL uploadSuccess, NSString *message, NSString *reportId) {
            NSLog(@"[Coverage] 上传%@: %@", uploadSuccess ? @"成功" : @"失败", message);
        }];
    }
}

- (void)applicationWillTerminate:(UIApplication *)application {
    if ([CoverageCollector dumpCoverageData]) {
        [CoverageUploader uploadLatestCoverageFileWithCompletion:nil];
    }
}

@end
```

**Swift（混编，需要先按第 5 节配置 Bridging Header）：**

## 8. 覆盖率报告生成

### 8.1 工具链说明

| 工具 | 用途 |


**方式一：使用 .app 包中的二进制文件**

### 8.4 使用 coverage_report.sh 一键生成


## 9. 增量覆盖率分析

### 9.1 工作原理

增量覆盖率分析只统计**本次代码变更**对应的覆盖率：

```
Git Diff → diffParser.rb → 变更行号映射 → 过滤 lcov 数据 → 增量 HTML 报告
```

### 9.2 diffParser.rb 使用

```bash
ruby gitdiff/utils/diffParser.rb \
    --diff-file=gitdiff.diff \
    --coverage-info-file=Demo.info
```

**参数说明：**

| 参数 | 必填 | 说明 |
|------|------|------|
| `--diff-file` | ✅ | Git diff 文件路径 |
| `--coverage-info-file` | ✅ | lcov 覆盖率数据文件路径 |

**白名单文件类型：** `.m`、`.mm`、`.swift`

### 9.3 完整增量分析流程

```bash
# 1. 获取当前 commit
CURRENT=$(git rev-parse --short=7 HEAD)

# 2. 生成 Git Diff（与上一个 commit 对比）
git diff HEAD~1 HEAD --unified=0 > gitdiff.diff

# 3. 转换覆盖率数据
xcrun llvm-profdata merge -sparse Demo.profraw -o Demo.profdata

# 4. 导出 lcov
MACHO=$(find ./MachOFiles -type f -exec file {} + | grep 'Mach-O' | head -n1 | cut -d: -f1)
xcrun llvm-cov export "$MACHO" -instr-profile=Demo.profdata -format=lcov > Demo.info

# 5. 解析增量覆盖率
ruby gitdiff/utils/diffParser.rb \
    --diff-file=gitdiff.diff \
    --coverage-info-file=Demo.info

# 6. 生成增量 HTML 报告
genhtml -o incremental_html Demo_gather.info --ignore-errors category

# 7. 打开报告
open incremental_html/index.html
```

## 10. 上传到覆盖率平台

平台支持两种上传方式，分别对应两个不同的接口：

| 方式 | 适用场景 | 接口 |
|------|---------|------|
| **自动化采集**（推荐） | 测试人员日常用 App 测试，App 自己退后台上传 `.profraw` | `POST /api/builds/:buildId/raw-coverage`（见第 6 节 `CoverageUploader`） |
| **手动上传** | CI/CD 流水线，本机/CI 已转换好 LCOV `.info` 文件 | `POST /api/upload/coverage` |

> ⚠️ 这两个接口**不能混用**：`/api/upload/coverage` 只接受已转换好的 LCOV `.info`（或 Cobertura
> XML），直接传未转换的 `.profraw`/`.profdata` 会被拒绝（报错提示用 `llvm-cov export` 先转换）。
> 原始 `.profraw` 只能走 `/api/builds/:buildId/raw-coverage` 这个专门接口，由服务端用
> `llvm-profdata`/`llvm-cov` 自动转换、合并。

### 10.1 自动化方式：创建 Build，App 用 commitHash 自动换 buildId

**Step 1：CI（或本机）编译出 `.ipa`，创建 Build**

> ⚠️ **`binary` 字段必须传完整的 `.ipa`，不接受裸 Mach-O 二进制。** 原因：组件大多以独立动态
> framework 形式集成（`pod 'X', :git => '...'`，CocoaPods 默认就是这种），覆盖率映射数据在
> 组件自己的二进制里（`Frameworks/X.framework/X`），不在主 App 二进制里——只传主二进制的话，
> 组件自己的文件永远不会出现在覆盖率报告里（这是真实踩过的坑，参见第 13 节 Q11）。平台收到
> `.ipa` 后会自动解压，找到主二进制（`ENABLE_DEBUG_DYLIB=YES` 时优先用
> `<ExecutableName>.debug.dylib`，不是瘦启动器）+ `Frameworks/` 下所有 framework 二进制，
> 一起传给 `llvm-cov export`。即使项目没有动态 framework 组件，统一传 `.ipa` 也没有副作用。

```bash
# 1. 编译并打包出 .ipa（Debug，带覆盖率插桩；同时第 6.0 节的 Run Script 会写好 CoverageInfo.plist）
xcodebuild build -workspace YourApp.xcworkspace -scheme YourApp -configuration Debug \
  -destination 'generic/platform=iOS' -derivedDataPath ./build

APP=./build/Build/Products/Debug-iphoneos/YourApp.app
mkdir -p ./ipa_build/Payload && cp -R "$APP" ./ipa_build/Payload/
(cd ./ipa_build && zip -rq ../YourApp.ipa Payload)

# 2. 创建 Build（buildKey 必须跟 CoverageInfo.plist 里写的一致，见第 6.0 节；
#    单仓库项目可以不传 buildKey，平台会自动用 commitHash 当默认值）
curl -X POST https://coverage-platform.internal/api/builds \
  -F "binary=@YourApp.ipa" \
  -F "projectId=<PROJECT_ID>" \
  -F "commitHash=$(git rev-parse HEAD)" \
  -F "buildKey=$(./compute_build_key.sh .)" \
  -F "branch=$(git rev-parse --abbrev-ref HEAD)" \
  -F "gitDiff=$(git diff <base_commit>..HEAD --unified=0 -- YourApp/)" \
  -F "componentRepos=$(./extract_component_repos.sh .)"
```

不需要把返回的 `data.id`（buildId）记下来配进 App——App 里配的是 `projectId`（见第 6.2 节），
构建身份已经在编译时通过 `CoverageInfo.plist` 自动带上了，首次上传时 `CoverageUploader`
会自己换出 buildId。

> 通过蒲公英分发的项目（`POST /api/builds/from-pgyer`）走的是同一套逻辑——平台直接下载蒲公英
> 上的 `.ipa`，自动提取主二进制和所有 framework 二进制，不需要额外配置。

**Step 2：把 App 分发给测试人员**

App 接入了 `CoverageUploader` 之后，测试人员正常用 App、退后台，覆盖率数据会自动上传到这个
commit 对应的 Build。同一个 `buildId` 可以反复接收多个测试人员的多次上传，平台会自动合并、
重新计算覆盖率（`rawUploadCount` 字段记录了已上传次数）；同一个 commit 被重新打包多次也没关系，
都会落到同一个 Build 上。

### 10.2 手动方式：CI 转换后上传

适合 CI/CD 流水线，单次运行测试、生成覆盖率、上传，不需要 Build/buildId 概念：

```bash
# 1. 合并 profraw → profdata
xcrun llvm-profdata merge -sparse Demo.profraw -o merged.profdata

# 2. 导出 LCOV
xcrun llvm-cov export <binary> -instr-profile=merged.profdata -format=lcov > coverage.info

# 3. 上传（注意：上传的是转换后的 coverage.info，不是原始 .profraw）
curl -X POST https://coverage-platform.internal/api/upload/coverage \
  -F "projectId=<PROJECT_ID>" \
  -F "platform=ios" \
  -F "commitHash=$(git rev-parse HEAD)" \
  -F "branch=$(git rev-parse --abbrev-ref HEAD)" \
  -F "file=@coverage.info" \
  -F "gitDiff=$(git diff <base_commit>..HEAD --unified=0 -- YourApp/)"
```

## 11. API 参考

### CoverageCollector（Objective-C）

```objc
@interface CoverageCollector : NSObject

// 初始化（建议在 application:didFinishLaunchingWithOptions: 中调用）
// 使用 dispatch_once 保证只初始化一次
+ (void)initializeCollector;

// 保存覆盖率数据到 .profraw 文件
// 调用 __llvm_profile_write_file()
// @return YES 成功, NO 失败
+ (BOOL)dumpCoverageData;

// 获取当前覆盖率文件路径
// 格式: <Documents>/Coverage/<bundleID>_<yyyyMMdd_HHmmss>.profraw
// @return 文件路径字符串
+ (nullable NSString *)coverageFilePath;

// 获取所有 .profraw 文件列表
// @return 文件路径数组
+ (NSArray<NSString *> *)allCoverageFiles;

// 清除所有覆盖率数据文件
+ (void)clearCoverageData;

// 设置自定义输出目录（需在 initializeCollector 之前调用）
// @param directory 目录路径
+ (void)setCoverageDirectory:(NSString *)directory;

@end
```

> Swift 项目里调用方式完全一样（`CoverageCollector.initializeCollector()` /
> `CoverageCollector.dumpCoverageData()` / ...），通过第 5 节的 Bridging Header 混编调用，
> 没有独立的 Swift API。

### CoverageUploader

```objc
@interface CoverageUploader : NSObject

// 初始化（幂等，多次调用只生效一次）。commitHash 自动从 CoverageInfo.plist 读取，不需要传
+ (void)initWithBaseURL:(NSString *)baseURL projectId:(NSString *)projectId;

// 是否已初始化（不代表已经拿到 buildId——buildId 是首次上传时才异步解析的）
```

## 12. 平台接口规范

### 12.1 上传覆盖率数据（手动方式，需先转换成 LCOV）

> ⚠️ **此接口只接受 LCOV `.info` 格式**，不支持直接上传 `.profraw`/`.profdata` 原始数据
> （会被拒绝，报错提示先用 `llvm-cov export` 转换）。
> SDK 自动上传 `.profraw` 走的是另一个接口：`POST /api/builds/:buildId/raw-coverage`（见 12.2）。

```
POST /api/upload/coverage
Content-Type: multipart/form-data

表单字段：
  file         : File    LCOV 覆盖率报告文件（.info）
  projectId    : String  项目 ID
  platform     : String  固定 "ios"
  commitHash   : String  Git commit hash
  branch       : String  Git 分支名
  gitDiff      : String  git diff 文本内容（可选，用于增量覆盖率分析）

成功响应 (200):
{
  "success": true,
  "message": "Upload successful",
  "reportId": "report_abc123"
}

失败响应:
{
  "success": false,
  "message": "错误描述"
}
```

### 12.2 创建 Build / 上传原始覆盖率数据（自动化方式）

详见 [10.1 自动化方式：创建 Build，App 用 commitHash 自动换 buildId](#101-自动化方式创建-buildapp-用-commithash-自动换-buildid)。
简要接口形态：

```
POST /api/builds                          创建 Build
                                            必填：binary（Mach-O 二进制）、projectId、commitHash、branch
                                            可选：gitDiff、buildKey（不传默认等于 commitHash；
                                            CocoaPods + git 源组件化项目传 compute_build_key.sh 算出的复合指纹，
                                            见第 6.0 节）、componentRepos（JSON 数组字符串，
                                            extract_component_repos.sh 的输出，拉源码时备查用）

                                            同一个 (projectId, buildKey) 重复调用会复用已有 Build，
                                            不会产生新的 buildId（覆盖旧二进制，重置已合并的报告）。
                                            commitHash 始终是壳工程真实 commit，只用于源码拉取/展示，
                                            不参与去重判断——去重判断按 buildKey

GET  /api/builds/resolve                   SDK 用 ?projectId=&commitHash= 换 buildId（首次上传时自动调用；
                                            参数名仍叫 commitHash 是为了兼容现有 SDK，实际匹配的是 buildKey）
                                            404 表示这个构建身份还没创建过 Build

POST /api/builds/:buildId/raw-coverage     上传 .profraw（SDK 自动调用，服务端自动合并多次上传）
```

## 13. 常见问题

### Q1: `__llvm_profile_write_file` 返回非零值

**原因**：LLVM Profile 运行时未正确初始化或文件路径无效。

**解决**：
1. 确保在调用 `dumpCoverageData` 之前已调用 `initializeCollector`
2. 检查 Documents 目录是否有写入权限
3. 确认编译参数 `-fprofile-instr-generate` 已正确添加

### Q2: `.profraw` 文件生成为 0 字节

**原因**：App 没有执行到任何被插桩的代码。

**解决**：
1. 确保 App 有实际的操作交互
2. 检查编译参数是否正确应用到所有源文件
3. 尝试在模拟器上运行（模拟器权限更宽松）

### Q3: `llvm-cov export` 报错 "Failed to load coverage"

**原因**：二进制文件与 `.profraw` 文件不匹配（非同一构建产物）。

**解决**：
1. 确保使用**同一次构建**的 `.app` 二进制文件
2. 如果重新编译了 App，需要重新收集 `.profraw` 数据
3. 检查二进制文件是否包含调试符号（不应是 stripped 版本）

### Q4: 增量报告中没有数据

**可能原因**：

1. Git diff 中变更的文件类型不在白名单（`.m`、`.mm`、`.swift`）中
2. lcov 文件中的路径与 diff 文件路径不匹配
3. 变更的代码没有被执行到

**排查**：

```bash
# 检查 diff 文件内容
cat gitdiff.diff | grep "+++"

# 检查 lcov 文件中的源文件列表
grep "^SF:" Demo.info

# 手动运行 diffParser 查看日志
```

### Q6: 如何支持 SwiftUI 项目？

SwiftUI 使用 `@main App` 结构体而非 `AppDelegate`，可通过以下方式集成：

```swift
import SwiftUI

@main
struct MyApp: App {
    init() {
        CoverageCollector.initialize()
    }

    var body: some Scene {
        WindowGroup {
            ContentView()
        }
    }
}
```

对于进入后台 / 终止的监听，使用 `ScenePhase` 环境变量：

```swift
@main
struct MyApp: App {
    @Environment(\.scenePhase) var scenePhase

    init() {
        CoverageCollector.initialize()
    }

    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .onChange(of: scenePhase) { newPhase in
            switch newPhase {
            case .background:
                CoverageCollector.dumpCoverageData()
            default:
                break
            }
        }
    }
}
```

### Q9: Release 包中是否包含覆盖率代码？

如果只在专用的 `Debug-Coverage` Configuration（见 3.3 节）添加编译参数，`Debug`/`Release` 包中
**不会**包含覆盖率插桩代码，也不影响包体积和性能。`CoverageCollector.m` 本身（`extern` 声明 +
调用代码）会一直存在于二进制里，但如果对应的 LLVM Profile 运行时函数没有被链接进来（没插桩），
这几个函数调用在链接阶段会报 "Undefined symbols"——所以正式发布给用户的 Release 包**不能**直接复用
插桩过的 Configuration，必须用没有插桩参数的 `Release` Configuration 单独编译。

### Q10: 组件升级了但壳工程没提交代码，覆盖率报告是不是用错二进制了？

如果用的是 `compute_build_key.sh`（见 6.0 节）而不是直接拿壳工程 commit hash 当身份，就不会。
`compute_build_key.sh` 会解析 `Podfile.lock` 的 `CHECKOUT OPTIONS` 段，把所有 git 源组件锁定的
commit 也编码进 `buildKey`——组件版本变了，`buildKey` 就会变，平台会当成一次新的构建处理，不会
跟旧组件版本的二进制和报告混在一起。如果你看到的还是混用情况，先确认：1）组件确实是
`pod 'X', :git => '...'` 这种 git 源依赖（走私有 podspec 仓库、不带 `:git` 的依赖目前算不出
组件 commit，只能退化成普通 commit hash）；2）CI 和 Xcode Build Phase 是不是真的都在用同一份
`compute_build_key.sh`，没有某一边还在用旧的 `git rev-parse HEAD`。

### Q11: 组件文件完全不出现在覆盖率报告/文件列表里

**原因**：组件是以独立动态 framework 形式集成的（CocoaPods 默认就是这种，`pod 'X', :git => '...'`
不加 `:modular_headers`/静态库设置的话编译出来就是动态库）。覆盖率映射数据在组件自己的二进制
里（`Frameworks/X.framework/X`），不在主 App 二进制里——只传主 App 二进制给平台的话，
`llvm-cov` 根本看不到组件自己的源码，组件文件不会出现在文件列表里，不是覆盖率低、是完全没有
这个文件的记录。

**解决（结构层面：让组件文件出现在报告里）**：`POST /api/builds` 现在统一要求传整个 `.ipa`
（不接受裸二进制），平台会自动解压、把主二进制和 `Frameworks/` 下所有 framework 二进制一起传给
`llvm-cov export`。具体见
[10.1 节](#101-自动化方式创建-buildapp-用-commithash-自动换-buildid)的说明和示例。

**⚠️ 但这只解决了"看不看得到"，解决不了"覆盖率数字一直是 0%"**：动态 framework 在运行时是
独立的 Mach-O 镶像，有自己单独一份 LLVM profiling 计数器存储——`CoverageCollector` 调的
`__llvm_profile_write_file()` 只能刷写自己所在镶像（主 App）的计数器，永远碰不到动态
framework 自己镶像里那份，**不管组件代码有没有被执行，数字都会是 0%**。传 `.ipa` 只是让
`llvm-cov` 能识别这个文件/函数的结构（source mapping），采集不到真实执行次数。

**真正解决（让数字也是真的）：把组件编译成静态库**，在 Podfile 里加（`$static_framework`
数组填组件名）：

```ruby
$static_framework = ['FlipBook']  # 改成需要统计真实覆盖率的组件名

pre_install do |installer|
  installer.pod_targets.each do |pod|
    if $static_framework.include?(pod.name)
      def pod.build_type
        Pod::BuildType.static_framework
      end
    end
  end
end
```

`pod install` 之后这个组件的代码会在链接期直接合并进主二进制（跟 `nm` 查得到的
`___profc_.../___profd_...` 计数器符号一样，都进了主二进制），运行时只有一个镶像，
跟主 App 共用同一份计数器存储，覆盖率数据就能真实统计到了——已经用真实组件验证过，从
`0%` 变成了能反映"只测了部分分支"的真实数字（比如 `66.67%`）。组件如果有自己的资源文件，
`static_framework` 仍然保留资源打包能力，不影响这部分。
